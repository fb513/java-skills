# 基于设计文档的 Java 后端开发

基于技术设计文档目录，按依赖关系完成 Java 后端接口实现。优先保证实现符合设计、项目习惯和可验证性，不把所有任务都升级成重型流程。

---

## 执行流程

### Phase 1：分析与规划

1. 读取 `00-main-design.md` 和所有切片文件，理解实体、接口、依赖关系和验收标准
2. 如果存在 `reports/verification.md`，读取其中的测试设计、已知风险和人工验证项
3. 按依赖拆分实现顺序：
   - 基础层：Entity、Mapper、公共 Model
   - 接口层：Controller + Service
   - 验证层：单元测试、接口测试或最小可行验证
4. 对小任务直接执行；对多切片、跨模块或用户要求留档的 story，创建或更新 `docs/plans/<story>/reports/dev-progress.md`

`dev-progress.md` 只记录范围、进度、验证结果和风险，不维护冗长状态表。

### Phase 2：实现

先完成基础层，再实现接口层。共享同一文件或存在明确依赖的改动串行处理；互不影响的接口可并行推进。

实现时遵守：

- 不引入与项目风格冲突的新抽象
- 不自动提交代码，除非用户明确要求
- 发现设计缺口时先记录并向用户确认
- 测试按风险补充，不机械为每个实现点生成测试

### Phase 3：验证

运行与改动风险匹配的最小验证，例如：

- `mvn clean compile`
- 相关单元测试
- `cd api-tests && python -m pytest --collect-only`
- 用户指定的接口测试

完成后调用或遵循 `java-coding-review` 做编译和 diff 审查。已有接口测试时，可建议继续调用 `/api-test-verify` 做完整接口验证。

---

## 并行原则

- 操作同一文件的接口放在同一批次处理，避免写冲突
- 有明确依赖的接口串行执行
- 独立接口尽量并行

---

## 输出

最终回复简要列出实现内容、修改文件、验证命令和结果、剩余风险，以及 `dev-progress.md` 路径（如已生成）。
