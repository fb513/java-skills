# 后端技术设计生成

根据 PRD 文档生成后端技术设计：主设计文档 + 垂直切片文件。设计要足够指导实现和测试，但避免预设过重流程。

---

## 输入

用户提供 PRD 文档路径（如 `docs/prd/20260417-后端技术需求.md`）。如果用户未指定 PRD 路径，主动询问。

---

## 约束

1. **不生成代码** — DDL 建表语句除外
2. **不确定就问** — 遇到不清楚的业务逻辑，停下来向用户提问
3. **验收标准必须可验证** — 每个切片的验收标准要能被自动化测试或人工验收直接使用
4. **不创建空报告** — 报告由后续开发、Review 或验证阶段按需生成

---

## 执行流程

### Phase 1：探索项目 + 分析 PRD

探索现有代码，理解项目现状。重点关注：

- 已有哪些表和实体可以复用
- 已有哪些本地数据可以直接使用
- 项目的命名惯例、包结构和接口风格

通读 PRD，如有不确定的点，整理问题一次性向用户确认。

### Phase 2：生成主设计文档

输出路径：`docs/plans/YYYYMMDD-<topic>/00-main-design.md`

文档结构：

```markdown
# <功能名称> — 主设计文档

> 基于 PRD `<PRD路径>`，面向后端开发。

## 1. 核心实体
列出实体，区分新建和复用。

## 2. 字段梳理
列出字段、枚举、索引和业务约束。

## 3. 建表 DDL
完整 CREATE TABLE 语句。

## 4. 新增 Java 包结构
按分层列出新增的类文件。

## 5. 切片总览
| # | 切片名称 | 依赖 | 涉及接口 |

## 6. 设计决策说明
关键架构选择及理由。

## 7. 验证策略
优先覆盖的接口测试、回归风险和人工确认项。
```

### Phase 3：生成切片文件

每个切片一个文件：`docs/plans/YYYYMMDD-<topic>/NN-slice-<name>.md`

切片文件结构：

```markdown
# Slice NN: <切片名称>

| 依赖 | 交付物 |
|------|--------|

## 涉及接口

### POST /api/xxx/create

**认证方式**: JWT

**请求 / 响应**: JSON 示例 + 字段说明

**实现逻辑**: 按步骤描述业务流程。

**错误场景**: 表格列出场景、code、说明。

## 验收标准

- [ ] 正常路径：...
- [ ] 权限 / 角色要求：...
- [ ] 参数边界：...
- [ ] 错误场景：...
- [ ] 状态流转 / 兼容性要求：...
- [ ] 编译通过，无新增警告
```

验收标准必须具体到可观察行为，避免只写“功能正常”“返回正确”等不可验证描述。没有对应风险的项可以省略，不必机械凑齐。

### Phase 4：用户确认

展示产出，确认设计决策、切片粒度和验收标准，迭代修改直到用户认可。

---

## 下一步

技术设计确认后，按任务复杂度选择后续流程：

1. `/api-test-generation`：需要先定义接口测试或生成测试骨架时使用
2. `/java-development`：基于设计文档实现接口
3. `/java-coding-review`：实现后做编译和 diff 审查
4. `/api-test-verify`：已有接口测试后运行验证和问题归因
