# 接口测试验证与修复

运行 pytest 验证测试是否通过。失败时以设计文档为判定依据，循环修复直到全部通过。

---

## 输入

用户提供**设计文档目录**（如 `docs/plans/20260501-xxx/`）— 作为行为判定的唯一权威来源。

测试运行范围：全量运行 `api-tests/` 下所有测试，既验证新测试也发现回归。

---

## 核心原则：设计文档是裁判

| 情况 | 判定 | 动作 |
|------|------|------|
| 代码行为与设计文档不符 | 代码 bug | 修代码 |
| 测试期望与设计文档不符 | 测试写错 | 修测试 |
| 设计文档未明确定义 | 灰色地带 | 向用户确认 |
| 环境问题（服务未启动等） | 环境问题 | 报告给用户，不修改代码 |

**绝对禁止**：为了让测试通过而随意修改代码行为。每次修代码前必须引用设计文档中的具体条目作为依据。

---

## 执行流程

1. 读取设计文档，理解每个接口的预期行为
2. 确认服务在运行（`curl -s http://localhost:8088` 或配置中的 base_url）
3. 全量运行测试：`cd api-tests && python -m pytest -v --tb=long 2>&1`
4. 对每个失败用例，对照设计文档判定改哪边，修复
5. 修完重跑，循环直到全部通过
6. 汇总报告：改了什么、为什么改

---

## 熔断

- 同一测试连续失败 3 轮 → 停止，向用户报告
- 修复循环超过 5 轮 → 停止，汇总当前状态
- 环境问题 → 立即停止

---

## 已有测试回归

新代码导致已有测试失败时，优先修复新代码使其兼容既有行为（已有测试是经过验证的行为基准），除非设计文档明确要求改变该行为。
