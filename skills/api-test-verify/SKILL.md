# 接口测试验证

运行 pytest 验证接口行为是否符合设计文档。纯验证请求只做验证和归因；执行型请求进入修复模式。

---

## 输入

用户提供设计文档目录（如 `docs/plans/20260501-xxx/`）。设计文档是判定行为对错的主要依据。

测试运行范围默认是 `api-tests/` 全量测试；用户指定范围时按指定范围执行。

## 模式判断

- 验证模式：用户说“验证 / 跑一下 / 生成报告 / 看失败原因”，只运行测试、归因并报告
- 修复模式：用户说“跑通 / 修复失败 / 直到通过 / 继续开发自检”，修复归因明确的问题并重跑验证

---

## 核心原则

| 情况 | 判定 | 默认动作 |
|------|------|----------|
| 代码行为与设计文档不符 | 代码 bug | 报告；修复模式下修代码 |
| 测试期望与设计文档不符 | 测试问题 | 报告；修复模式下修测试 |
| 设计文档未明确定义 | Spec 灰区 | 停下来向用户确认 |
| 服务、配置或外部依赖不可用 | 环境问题 | 报告环境阻塞 |

不要为了让测试通过而随意修改业务行为或断言。修复前必须能指向设计文档、既有行为基准或用户确认。

---

## 执行流程

### Phase 1：收集上下文

1. 读取设计文档和相关切片
2. 如存在 `reports/verification.md` 或 `reports/dev-progress.md`，读取其中的测试设计、已知风险和人工验证项
3. 判断本次是验证模式还是修复模式

### Phase 2：确认环境

确认服务、配置和依赖可用。服务未启动、数据库/Redis 不可用或 base_url 配置错误时，作为环境问题报告，不修改代码或测试。

### Phase 3：运行测试

默认命令：

```bash
cd api-tests && python -m pytest -v --tb=long
```

记录测试统计、失败用例、跳过原因和关键请求响应。

### Phase 4：归因与记录

对每个失败用例归类为：

- code：实现偏离设计文档或既有兼容性要求
- test：测试断言、fixture 或前置条件与设计文档不一致
- spec：设计文档缺失、含糊或自相矛盾
- env：服务、配置、数据源或外部依赖问题

复杂 story 或用户要求留档时，更新 `docs/plans/<story>/reports/verification.md`，只记录必要信息：测试命令和统计、失败归因、建议动作、人工验收清单。小范围验证直接在最终回复中说明即可。

### Phase 5：修复模式

仅修复归因清楚的问题：

- code：修 Java 代码
- test：修 pytest 测试
- spec：停止并询问用户
- env：停止并报告环境阻塞

修复后重跑相关测试；必要时重跑全量测试。同一问题连续失败 3 轮时停止并汇报。

---

## 输出

最终回复简要列出测试命令和结果、各类问题数量、已修复内容、仍需用户确认的问题，以及 `verification.md` 路径（如已生成）。
